# Azure CI/CD and Training Trigger Plan

This document describes the planned Azure automation for the project. It covers
two related but separate areas:

- application CI/CD: build and deploy the frontend/API containers to Azure
  Container Apps;
- training automation: decide when Azure ML should run the preprocessing and
  training pipeline.

## Current State

The application deployment chain already exists:

```text
push to main
  -> Code Quality
  -> Docker Build & Push
  -> Deploy to Azure Container Apps
```

The training trigger policy also already exists in code:

- `src/sign_language_training/orchestration/trigger_policy.py`
- `src/sign_language_training/orchestration/dataset_inventory.py`
- `src/sign_language_training/orchestration/training_state.py`
- `scripts/trigger_training.py`
- `scripts/check_data_change_and_train.py`
- `src/sign_language_training/trigger_api/app.py`

The policy currently supports:

- `manual`: submit only when forced;
- `data_change`: submit when enough images changed;
- `scheduled`: submit when the previous submission is old enough.

The missing part is Azure scheduling: nothing currently creates an Azure ML
schedule that runs the trigger checker automatically.

## Target Trigger Policy

The project target is:

```text
Run a daily checker.

If at least 10 new images exist:
    submit Azure ML preprocessing + training pipeline.
Else if the last submitted training is older than 7 days:
    submit Azure ML preprocessing + training pipeline.
Else:
    skip training.
```

This means Azure ML should not directly run training every day. Azure ML should
run a small checker job every day, and the checker decides whether the expensive
training pipeline should start.

Required default thresholds:

```text
TRAINING_TRIGGER_MIN_NEW_IMAGES=10
TRAINING_TRIGGER_INTERVAL_DAYS=7
```

The existing setting name `TRAINING_TRIGGER_MONTHLY_INTERVAL_DAYS` is misleading
for weekly retraining. Rename it immediately instead of carrying the confusing
name forward:

```text
TRAINING_TRIGGER_INTERVAL_DAYS
```

Do not keep `TRAINING_TRIGGER_MONTHLY_INTERVAL_DAYS` as a compatibility alias.
New code and documentation should use `TRAINING_TRIGGER_INTERVAL_DAYS`.

## Planned New Script

Add:

```text
scripts/create_training_schedule.py
```

Purpose:

- create or update an Azure ML schedule;
- schedule a lightweight checker job, probably daily;
- pass trigger thresholds and dataset settings into that checker job;
- avoid manual schedule setup in `ml.azure.com`.

The scheduled checker should not call only:

```bash
python scripts/trigger_training.py --reason scheduled
```

That checks only the time condition. We need both conditions. Better options:

1. Add a new `auto` trigger reason.
2. Add a separate checker script that evaluates `data_change` first, then
   `scheduled`.

Preferred approach:

```text
scripts/check_training_triggers.py
```

Logic:

```text
evaluate data_change with min_new_images=10
if submitted:
    stop

evaluate scheduled with interval_days=7
if submitted:
    stop

print skip decision
```

This avoids overloading the existing `scheduled` reason.

## Blobstore and Training Data

The Azure ML workspace has these datastores:

- `workspaceblobstore` - default Azure Blob Storage datastore;
- `workspaceartifactstore` - stores run artifacts and outputs;
- `workspaceworkingdirectory` - Azure file share working directory.

Training data lives in `workspaceblobstore`.

Training outputs and registered model artifacts are produced by Azure ML jobs.
The trained model is registered in the Azure ML `Models` tab as
`ngt-sign-language`. If a newly trained model passes evaluation and is selected
as the production candidate, it receives the `promoted` tag.

Current `ngt-raw` data is stored under an Azure ML generated upload path:

```text
azureml://subscriptions/0a94de80-6d3b-49f2-b3e9-ec5818862801/resourcegroups/buas-y2/workspaces/team-R2-2026/datastores/workspaceblobstore/paths/LocalUpload/fa6a4d116f677a8c255e4784df45d295a2ec3d8fcd795c07832cf8b4e2519e81/raw/
```

This works, but it should not become the long-term project convention.
`LocalUpload/<hash>/...` is generated by Azure ML when data is uploaded from a
local directory. It is not human-friendly and is awkward for future automation.

Recommended canonical raw-data path:

```text
azureml://datastores/workspaceblobstore/paths/datasets/ngt/raw/current/
```

Fully qualified equivalent for this workspace:

```text
azureml://subscriptions/0a94de80-6d3b-49f2-b3e9-ec5818862801/resourcegroups/buas-y2/workspaces/team-R2-2026/datastores/workspaceblobstore/paths/datasets/ngt/raw/current/
```

Optional supporting paths:

```text
azureml://datastores/workspaceblobstore/paths/datasets/ngt/raw/batches/
azureml://datastores/workspaceblobstore/paths/datasets/ngt/raw/manifests/
```

## Data Versioning Decision

Question:

Should each new `ngt-raw` version contain only the new batch, or should each new
version represent the full training dataset?

Recommendation:

Each `ngt-raw` version should represent the full dataset that training should
use.

Reason:

The current training pipeline accepts one raw ImageFolder input:

```text
azureml:ngt-raw:<version>
```

It does not currently load and merge many versions of `ngt-raw`. If version `5`
contains only the newest 10 images, then training on `azureml:ngt-raw:5` trains
only on those 10 images. That is wrong for normal retraining.

So the safe semantic rule is:

```text
ngt-raw version N = complete raw dataset snapshot for training run N
```

Manual registration flow:

```text
1. Upload only the new images into:
   datasets/ngt/raw/current/<class>/

2. Register a new `ngt-raw` version pointing to:
   azureml://datastores/workspaceblobstore/paths/datasets/ngt/raw/current/

3. Submit or allow the scheduled checker to submit training with:
   azureml:ngt-raw:<new-version>
```

This means old files are not uploaded again one by one. The canonical `current`
folder already contains the old data, and only new images are added to it.

Tradeoff:

Using a mutable `current` folder is cheaper and simpler, but weaker for strict
reproducibility. If later files are changed under `current`, an old registered
asset version pointing to that path may no longer represent exactly the same
file contents. The raw data registration should therefore store metadata tags
such as `image_count`, `manifest_hash`, and `registered_at`.

Cost concern:

Registering every version as a full copied folder can duplicate data and cost
more storage. To reduce this, use an append-only canonical dataset location in
`workspaceblobstore`, for example:

```text
azureml://datastores/workspaceblobstore/paths/datasets/ngt/raw/current/
```

Then add new images into that canonical location and register a new `ngt-raw`
version pointing to the current canonical dataset path. Ideally, registration
should point to blob storage instead of re-uploading a local copy every time.

This may require extending:

```text
scripts/register_raw_data.py
```

Possible extension:

```text
poetry run python scripts/register_raw_data.py \
  --data-uri azureml://datastores/workspaceblobstore/paths/datasets/ngt/raw/current/ \
  --data-dir data/raw \
  --asset-name ngt-raw \
  --version 6
```

Batch-only versions are not recommended unless a future pipeline merge step is
added that combines all historical batch assets before preprocessing. That is
more complex and easier to get wrong.

## Preprocessing and Processed Data

The current Azure ML pipeline starts from raw data and preprocesses during the
training pipeline run:

```text
ngt-raw
  -> preprocessing step
      -> stratified split
      -> train augmentation
      -> validation/test outputs
  -> training step
      -> model registration
```

The team does not manually maintain processed data. Azure ML still stores
pipeline step outputs internally so that the training step can consume the
preprocessing outputs, and `augmentation.py` can register `ngt-augmented-train`
with a `ngt_raw_version` tag. However, cache reuse is currently disabled in
`pipeline_submitter.py`, so the practical behavior is fresh preprocessing for
full retraining runs.

## Trigger State Without Blob Writes

The local/API trigger policy still stores state in:

```text
state/training_trigger_state.json
```

That is fine for local/manual runs, but Azure ML scheduled jobs run on temporary
compute and the team does not currently have storage-level blob write
permissions needed for a shared JSON state file or blob lease.

The Azure scheduled checker should therefore use Azure ML metadata as the
source of truth:

- latest dataset state: newest `ngt-raw` data asset version and its tags;
- last training time: latest completed Azure ML retraining job;
- previous image count: `trigger_image_count` tag on that retraining job;
- active duplicate protection: current active Azure ML retraining jobs.

Raw `ngt-raw` versions should include:

- `image_count`;
- `manifest_hash` when known;
- `registered_at`.

Submitted retraining pipeline jobs should include:

- `purpose=retraining`;
- `raw_data_asset`;
- `raw_data_version`;
- `trigger_reason`;
- `trigger_image_count`;
- `manifest_hash` when known.

## Concurrency and Duplicate Training Prevention

The checker must not submit two training pipelines if two scheduled checks start
at the same time.

A fixed delay, for example waiting 10 minutes between checks, reduces the chance
of overlap but does not solve the problem. Two jobs can still start close
together, and a training submission can take longer than expected.

Use the active Azure ML job check before submission.

Implementation concept:

```text
daily Azure ML schedule starts checker job
checker reads latest ngt-raw version metadata
checker reads latest completed retraining job metadata
checker checks Azure ML for active retraining jobs

if an active retraining job exists:
    skip
elif latest_raw.image_count - last_job.trigger_image_count >= 10:
    submit training
elif last_training > 7 days:
    submit training
else:
    skip
```

This does not fully eliminate the theoretical race where two checker jobs start
at the exact same second before either submitted job appears in Azure ML. For a
daily schedule this risk is small and the simpler design fits the current
permissions better. If duplicate jobs ever happen in practice, add deterministic
job naming as the next protection.

## CI/CD Smoke Tests

Add post-deploy smoke tests to `.github/workflows/deploy-azure.yml`.

After Container Apps deploy, automatically check:

```text
backend /info returns 200
backend /health returns 200 if available
frontend returns 200
WebSocket /ws/predict accepts a frame
WebSocket response includes hands[].landmarks with 21 points
```

This is important because a GitHub Actions deployment can be technically
successful while the app is still broken. The previous Azure issues with backend
routing and missing MediaPipe landmarks would have been caught by this.

## Manual Workflow Dispatch

Add `workflow_dispatch` to the Azure deployment workflow.

This means GitHub Actions gets a manual "Run workflow" button. It lets us deploy
or redeploy without pushing a new commit.

Useful manual inputs:

```text
image_tag
deploy_ml_endpoint
model_version
deployment_name
backend_revision
frontend_revision
```

Example use cases:

- redeploy the same image after fixing Azure variables;
- deploy a specific model version;
- skip Azure ML endpoint deployment and only redeploy Container Apps;
- test a new frontend/backend image tag;
- roll back to a known good revision.

This is not a replacement for automatic CI/CD. It is an operational control for
debugging and recovery.

## Rollback

Rollback means returning production to a previous known-good version after a bad
deployment.

There are two rollback levels in this project.

### Container App Rollback

Azure Container Apps keeps revisions. Every time the workflow updates the image
or environment variables, Azure creates a new revision.

If a new revision is broken, rollback means activating an older revision or
sending traffic back to it.

Example:

```powershell
az containerapp revision list `
  --name signlang-r2-api `
  --resource-group buas-y2 `
  --output table

az containerapp revision activate `
  --name signlang-r2-api `
  --resource-group buas-y2 `
  --revision <previous-good-revision>
```

This affects the app containers only.

### Azure ML Model Rollback

Azure ML online endpoints can use blue/green deployments. If the `green` model
deployment is bad, rollback means sending traffic back to `blue`.

Example concept:

```text
blue  = previous good model
green = new candidate model

If green fails:
    set blue traffic to 100%
    set green traffic to 0%
```

The repo already has rollout helpers:

- `scripts/rollout_blue_green.py`
- `scripts/rollout_canary.py`

The Azure CI/CD workflow should eventually expose rollback through
`workflow_dispatch`, so rollback can be triggered from GitHub Actions without
typing Azure CLI commands manually.

9. Add rollback support to the Azure workflow.

## Decisions

Confirmed:

1. `ngt-raw` should mean the complete raw dataset used for training.
2. The checker should run daily as a starter.
3. The schedule should first be created manually with
   `scripts/create_training_schedule.py`; GitHub Actions integration can be
   added later.
4. The Azure scheduled checker should not write trigger state to
   `workspaceblobstore`; it should use Azure ML data asset and job metadata.
5. Duplicate checker runs should be prevented with the existing active Azure ML
   job check. Blob lease locking is intentionally not used because the current
   project identity lacks storage-level blob write permissions.

Still to confirm:

1. Whether to migrate current data from the generated `LocalUpload/<hash>/raw/`
   path into `datasets/ngt/raw/current/`.
2. Whether strict dataset reproducibility is required. If yes, use immutable
   snapshot folders such as `datasets/ngt/raw/snapshots/v006/` instead of a
   mutable `current/` folder.
